SFT+GRPO V2 改进方案

Executive Summary

本轮 SFT 后继续 GRPO 没有超过 SFT，主要不是因为格式、推理环境或评测脚本坏了，而是因为 GRPO 把模型推向了“少报 turn、保留高置信长片段”的策略。

HarperValley 上的直接症状是：

模型OverallTurn PrecisionTurn RecallTurn F1Speaker Acc.WERTemporal IoU
SFT85.0190.9590.5090.7394.208.1463.24
SFT+GRPO79.2593.9446.5062.2194.6212.9673.15

GRPO 确实修掉了一些 SFT 的表面问题：HarperValley repetition flags 从 86/100 降到 15/100，completion tokens 中位数从 182.5 降到 106.0，Temporal IoU 从 63.24 提到 73.15。但它同时把平均 segment 数从 3.98 压到 1.98，而 HarperValley reference 平均是 4 turns。最终 recall 崩掉，overall 被拉低。

结论：下一步不应先靠单纯调学习率或多训 epoch。先要改 reward / curriculum，让漏 turn 的代价足够大，并在训练中显式监控 predicted-turn count 和 recall。

发生了什么

1. SFT+GRPO 学到了“短输出”策略

诊断项SFTSFT+GRPO
HarperValley 有效解析100/100100/100
重复输出标记86/10015/100
completion tokens 中位数182.5106.0
平均预测 segments3.981.98
预测 segment 数直方图2:14, 3:24, 4:27, 5:25, 6+:101:6, 2:90, 3:4

这能很直接地解释指标变化：

  • Precision 上升，是因为模型输出的 turn 更少。
  • Temporal IoU 上升，是因为保留下来的 turn 往往是最容易、最长、对齐最好的片段。
  • Recall 崩掉，是因为后面的短回应、backchannel 和收尾 turn 被省略了。
  • WER 小幅变差，是因为一些被遗漏的短 turn 被合并进相邻的长 turn。

2. 局部样例显示的问题是省略和合并，而不是随机退化

样例 hvb_0020：

  • SFT+GRPO 只输出前两个 turn，而且这两个几乎完全正确。
  • SFT 输出了五个 turn，包括网点营业时间回答和收尾 turn。
  • SFT+GRPO 在匹配到的 turn 上 Temporal IoU 是 0.985、WER 是 0.0，但 recall 只有 0.40。

样例 hvb_0035：

  • SFT+GRPO 把多个 speaker exchange 合并成一个很长的 Speaker 1 segment。
  • Parser 会接受这个输出，但 6 个 reference turns 里只有 1 个能匹配。

样例 hvb_0074：

  • SFT+GRPO 把 “that was everything” 合并进银行客服的回答里，并省略了后续短轮次结构。
  • 第一个匹配 turn 的 Temporal IoU 很高，但 turn recall 只有 0.1667。

这是一个典型的 reward shaping 失败：输出更干净、更短，但不完整。

训练证据

SFT+GRPO 确实是从下面的 SFT checkpoint 正确初始化的：

/data3/fwh/temporal-audio-caption/sft_harpervalley_v1/final

本次训练配置为：

  • num_generations=4
  • temperature=0.7
  • learning_rate=5e-7
  • beta=0.04
  • curriculum_steps=50
  • 训练一轮 / 167 个 optimizer steps
  • 使用相同的 data/harpervalley_grpo
  • 使用相同 prompt 和 parser
  • 使用与 grpo_harpervalley_v1 相同的 who/what/when reward functions

也就是说，常见的 SFT→GRPO 条件基本都具备：SFT 初始化、reference model 锚定同一个 checkpoint、KL penalty，以及 online sampled completions。

但训练曲线解释了为什么结果仍然失败：

Steps 40-600.7040.2100.0040.762170.9172.0

解读：

  • 在 SFT 初始化阶段，采样 completion 经常触发高重复的 zero-reward gate。
  • 于是 GRPO 找到了一条最容易避开重复惩罚的路径：显著缩短输出。
  • 当前 reward 通过 Turn F1 间接惩罚漏 turn，但力度不足，无法阻止“短而高 precision”的策略。
  • reward_std 在 SFT+GRPO 中更弱：167 steps 里有 46 steps 的 reward_std < 0.01，而 base GRPO 是 0/167。这说明很多 group 缺少有效的相对信号，尤其是在早期。

这只是参数问题吗？

有一部分是，但主要不是。

参数确实可能有影响：

  • beta=0.04 让训练后期 KL 升到约 0.08-0.10，高于原始 GRPO run 常见的 0.03-0.05。这使模型明显偏离了较强的 SFT policy。
  • 完整一轮训练已经足够让模型强烈转向短输出。
  • num_generations=4 对接近 SFT 的 policy 来说 group diversity 有限，许多早期 group 的 reward variance 接近 0。
  • temperature=0.7 可能在早期采样出重复的 SFT completions，触发 hard zero gate，并把模型推向更短的输出。

但单纯调参不太可能解决核心问题，因为 reward target 本身没有充分约束 turn coverage。更低 LR 或更高 beta 可能减少伤害，但不会教会模型“漏掉最后 2-4 个 turn”比“输出一个简洁干净答案”更糟。

常见 SFT+GRPO 条件哪些满足了？哪些没有充分满足？

已满足

  • 使用 SFT checkpoint 作为 policy 初始化。
  • Reference model 从同一个 model_name_or_path 加载，因此 KL anchor 是 SFT policy。
  • 复用了相同的 train/validation data 和 prompt。
  • Rule-based rewards 是确定性且可复现的。
  • Evaluation 使用同一个 HarperValley protocol。

未充分满足

  • Coverage reward 太弱。 漏 turn 的惩罚只通过 Turn F1 间接体现，而最终加权 reward 仍可能让两个干净匹配 turn 得到可接受分数。
  • 没有显式 turn-count reward / guard。 Trainer 没有单独记录或优化 predicted-turn count、recall、count error。
  • Hard repetition gate 在早期主导了 SFT+GRPO。 SFT eval 有很多 repetition flags，所以早期 sampled completions 常被置零。最容易的改进方向变成缩短输出，而不是更好地切分 turn。
  • 没有 validation checkpoint selection。 最终使用的是 final checkpoint。最佳 tradeoff 可能出现在模型强烈压缩输出之前。
  • 对 near-SFT policy 来说 group size 太小。 num_generations=4 在很多 steps 产生低 reward variance，削弱了 GRPO 区分“完整但偏啰嗦”和“短而干净”的能力。

为什么 MMAU/MMSU 也没有提升？

模型MMAU speechMMSU split
SFT40.5438.60
SFT+GRPO39.9437.80

Probe 结果和 HarperValley 的失败模式一致：

  • SFT+GRPO 在 probe captions 中显著减少了重复。
  • 但它也减少了细节和覆盖度，这会伤害 caption-QA。
  • MMAU 接近 SFT，是因为很多 speech questions 仍然受益于更干净、简洁的 captions。
  • MMSU 略降，可能是因为更广泛的 audio reasoning 更依赖被省略的上下文。

下一步建议

优先级 1：用 coverage-first reward 重新跑 GRPO

创建 sft+grpo_harpervalley_v2，先改 reward，再做大规模参数 sweep。

推荐的 reward 修改：

  • 加入显式 turn_recall_reward。
  • 加入显式 turn_count_reward = max(0, 1 - abs(predicted_count - reference_count) / reference_count)。
  • 把漏 turn 行为设成 gate 或强 multiplier：对于 4+ turns 的 clips，如果 recall < 0.8，则限制 total reward 上限。
  • 保留 repetition penalty，但避免让 repetition 成为早期主导信号。对边界重复模板，优先使用 graded repetition penalty，而不是立即置零。

建议最终权重：

组件权重
format0.10
turn recall / count0.30
who0.20
what0.20
when0.15
repetition/length quality0.05

重要：每一步都记录 predicted_turns、reference_turns、turn_recall、turn_precision、count_error 和 repetition_ratio。

优先级 2：使用更温和的 SFT+GRPO schedule

Reward 修好后，使用更保守的训练配置：

  • learning_rate=1e-7 或 2e-7
  • beta=0.08 或 0.10
  • 先用 max_steps=50，不要直接训完整 epoch
  • 如果显存允许，使用 num_generations=6 或 8
  • 保持 temperature=0.5-0.7；如果 repetition 仍然高，先用较低 temperature

如果 validation predicted-turn mean 低于 3.5，或者 recall 比 SFT 低超过 5 points，就提前停止。

优先级 3：在丢弃现有 run 前，先做 checkpoint sweep

当前保存的 checkpoint-167 和 final 都是后期 checkpoint，但 training log 显示短输出是在后期加剧的。下一次训练应每 25 或 50 steps 保存并评估：

  • checkpoint-25
  • checkpoint-50
  • checkpoint-75
  • checkpoint-100
  • checkpoint-125

最佳模型可能是一个中间 checkpoint：repetition 已经改善，但 turn recall 还没有崩掉。

优先级 4：在完整 probe eval 前增加小型 regression harness

在跑 MMAU/MMSU 之前，先评估 10-20 个 HarperValley validation/benchmark examples，并要求满足：

  • parse rate 100%
  • predicted-turn mean 接近 reference-turn mean
  • 没有系统性的 2-turn collapse
  • repetition flags 低于 SFT
  • 没有 finish_reason=length

这样可以在几分钟内发现本次失败模式。

推荐实验矩阵

实验改动目的
v2areward 加入 turn recall/count；LR 2e-7；beta 0.08；50 steps测试 coverage reward 能否防止 collapse

只有当短 run 保住 turn recall 后，才运行完整 HarperValley+MMAU+MMSU。

一句话结论

SFT+GRPO 失败并不是因为 “SFT then GRPO” 这条路线错了，而是因为当前 GRPO 设置奖励了更干净但不完整的 captioning policy。下一轮应该把 coverage 作为一等 reward component，并使用更短、更强 KL 约束的 post-SFT update。
