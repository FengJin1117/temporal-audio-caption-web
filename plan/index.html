时序化 GRPO Demo 计划 v1

一个“小而精、快准狠”的 dialogue speech temporal caption demo：用 D-ORCA 的 who / what / when 拆解，暴露并修复 Qwen2.5-Omni 的时序化 caption 问题。

Qwen2.5-Omni Dialogue Speech GRPO Speaker / Content / Time Reward 1000 条精选数据

1. Demo 定位

本 demo 不追求大而全，目标是用最小数据和最直接 reward 暴露并修复 Qwen2.5-Omni 在 dialogue speech caption 中的 temporal caption 问题。

核心任务从泛化 audio caption 收窄为：

输入一段含多轮对话的音频，输出句子级 temporal dialogue caption：
[start, end] speaker: transcript / concise utterance

这里借鉴 D-ORCA 的拆解：

Dialogue temporal caption = who + what + when
  • who：谁说的，或至少 speaker turn 是否区分正确。
  • what：说了什么，转写/语义是否准确。
  • when：每句话或每个 speaker turn 的时间边界是否对齐。

本 demo 的论文价值在于证明：对 dialogue speech 场景，时序化 caption 可以通过结构化输出和可验证 reward 被 GRPO 快速改善。

2. 输出格式

统一要求模型输出结构化 caption，便于 reward 解析。

<answer>
[00.00-02.10] Speaker 1: ...
[02.10-04.80] Speaker 2: ...
[04.80-07.20] Speaker 1: ...
</answer>

可选增强版：

<think>
#Turn 1: speaker=Speaker 1; time=[00.00,02.10]; content="..."
#Turn 2: speaker=Speaker 2; time=[02.10,04.80]; content="..."
</think>
<answer>
[00.00-02.10] Speaker 1: ...
[02.10-04.80] Speaker 2: ...
</answer>

demo v1 建议先只强制 <answer>，避免模型被复杂格式拖垮。后续若 reward 解析不稳，再加入 <think>。

3. 评测协议

3.1 评测目标

评测必须能暴露 Qwen2.5-Omni 的具体问题，而不是只给一个笼统 caption 分数。

  • speaker turn 漏掉或合并。
  • speaker attribution 错误。
  • transcript 只给摘要、不保留具体话语。
  • timestamp 缺失、顺序错乱、边界偏移。

3.2 主评测集

构造一个小型 Dialogue-TCap-Bench：

  • 规模：100 条音频。
  • 语言：优先英文 60 条，中文 40 条；如果中文处理成本高，v1 可先全英文。
  • 时长：8-30 秒。
  • 每条包含 2-6 个 speaker turns。
  • 场景：访谈、影视对白、播客片段、客服/会议、双人问答。
  • 标注字段：audio_path、turn_id、start、end、speaker_id、transcript。

优先选择 speaker turn 清晰、背景噪声不过强、没有大量重叠说话的样本。demo v1 不把 overlapped speech 作为重点。

3.3 指标

参考 D-ORCA 的 DVD-Bench，采用三主指标：

指标计算方式目的
Speaker Acc匹配后的 turn 中 speaker_id 正确比例衡量 who
WER / CER将预测 transcript 与 GT transcript 对齐后计算衡量 what
Temporal IoU预测 turn 时间段与 GT turn 时间段的平均 IoU衡量 when

补充指标：

指标计算方式目的
Turn F1预测 turn 与 GT turn 按时间 IoU + 文本相似度匹配后算 F1衡量是否漏 turn / 多 turn
Order Acc匹配 turn 的顺序是否保持一致暴露 temporal ordering 错误
Format Pass Rate是否可解析成 [start-end] speaker: text保证 reward 可计算
Length / Repetition Rate输出是否过长、重复 speaker 或重复句子防止 GRPO reward hacking

3.4 匹配规则

预测 turn 与 GT turn 做 bipartite matching。匹配分数：

match_score = 0.5 * temporal_iou + 0.5 * text_similarity
  • temporal_iou：预测 [start,end] 与 GT [start,end] 的 IoU。
  • text_similarity：v1 用 1 - normalized WER；后续可替换为 embedding similarity。

仅当 match_score >= 0.3 才视为有效匹配。

4. 数据制造计划

4.1 训练数据规模

GRPO 训练只做高质量 1000 条。

split数量用途
train1000GRPO
valid100reward sanity check
test100Dialogue-TCap-Bench

训练数据不追求覆盖所有音频场景，只服务于 dialogue temporal caption 行为塑形。

4.2 数据源优先级

优先级 A：真实 speech/dialogue

  • YouTube / podcast / interview clip：可用 WhisperX 或强制对齐生成句子级时间戳。
  • Movie / TV dialogue clips：适合 speaker attribution，但版权和角色标注成本较高。
  • Call / meeting / conversation datasets：适合 speaker turn 和 diarization。
  • D-ORCA / DVD-Bench 思路：若官方数据释放，可借鉴其 speaker + transcript + timestamp 格式；若未释放，只借鉴评测协议。

优先级 B：合成双人对话

  • 从公开文本对话生成 2-6 轮 script。
  • 用 TTS 生成 Speaker 1 / Speaker 2 两种音色。
  • 按脚本合成不同间隔、停顿、轻微重叠。
  • 自动保存每句 start/end/speaker/transcript。

demo v1 推荐以 B 为主、A 为辅：

700 条 TTS 合成双人对话
300 条真实 speech/dialogue 片段

原因：TTS 数据时间戳干净，reward 稳；真实数据用于防止模型只适应合成音色。

4.3 样本筛选

对候选样本先跑 Qwen2.5-Omni baseline，保留“可学习但没完全学会”的样本。

Format Pass = true
0.2 <= Speaker Acc <= 0.9
0.2 <= 1-WER <= 0.9
0.1 <= Temporal IoU <= 0.8
turn_count in [2, 6]
duration in [8s, 30s]

丢弃条件：

  • baseline 完全不会输出可解析格式。
  • 单人独白，无 speaker turn。
  • 大量重叠说话，连人工也难对齐。
  • transcript 过长，导致 Qwen 输出超长或重复。
  • 音频质量过差，ASR 本身不可靠。

4.4 数据字段

{
  "id": "demo_000001",
  "audio_path": "...wav",
  "duration": 18.4,
  "language": "en",
  "turns": [
    {
      "turn_id": 1,
      "speaker": "Speaker 1",
      "start": 0.24,
      "end": 2.10,
      "text": "..."
    }
  ],
  "prompt": "Generate a timestamped dialogue caption..."
}

speaker v1 不要求真实姓名，只要求 Speaker 1 / Speaker 2 一致。若使用影视或访谈数据，可额外保留 speaker_name，但 reward 先按匿名 speaker 计算。

5. Reward 设计

5.1 总 reward

借鉴 D-ORCA，reward 分成 who / what / when。

R = 0.15 * R_format
  + 0.35 * R_speaker
  + 0.30 * R_content
  + 0.20 * R_time

如果发现 timestamp reward 过早导致 collapse，使用 curriculum。

5.2 Format Reward

检查每行是否满足：

[start-end] Speaker k: text
  • 有 <answer>：+0.2。
  • 每行可解析 timestamp：+0.3。
  • 每行可解析 speaker：+0.2。
  • 每行有非空 text：+0.2。
  • 时间单调递增：+0.1。

若输出超过最大长度、重复率过高或无法解析：

R_total = 0

5.3 Speaker Reward

预测 turn 与 GT turn 匹配后计算 speaker 是否一致。

R_speaker = correct_speaker_count / matched_turn_count

如果使用匿名 speaker，需要先做 speaker permutation 对齐：

预测 Speaker 1 可以映射到 GT Speaker A 或 B，取全局最优映射。

5.4 Content Reward

对匹配 turn 的文本计算 WER/CER。

英文：R_content = max(0, 1 - WER)
中文：R_content = max(0, 1 - CER)

v1 可以把所有 matched turn 文本拼接后算一次 WER，稳定性更高；后续再改成 turn-level 平均。

5.5 Time Reward

对匹配 turn 计算 temporal IoU。

R_time = mean(IoU(pred_turn_i, gt_turn_j))

额外加入顺序惩罚：

若预测时间不单调或 turn 顺序反转，R_time *= 0.5

5.6 Curriculum

D-ORCA 的经验是：从一开始加入 time reward 可能导致训练崩溃。因此 demo v1 采用两阶段。

阶段步数reward
阶段 1前 30% step0.20 R_format + 0.45 R_speaker + 0.35 R_content
阶段 2后 70% step0.15 R_format + 0.35 R_speaker + 0.30 R_content + 0.20 R_time

如果 baseline 本身时间戳很差，可以先让模型输出离散时间：beginning / middle / end。但 demo v1 优先尝试秒级 timestamp，因为 dialogue turn 的边界比泛声事件更容易标注。

6. GRPO 训练设置

6.1 基模

主实验：Qwen2.5-Omni

可选对照：Qwen3-Omni-30B-A3B、Audio-Flamingo-Next。demo v1 不建议引入多基模，先把 Qwen2.5-Omni 的问题打穿。

6.2 Prompt

You are given an audio clip containing a short dialogue.
Generate a timestamped dialogue caption.
Use one line per speaker turn.
Each line must follow:
[start-end] Speaker k: spoken content
Do not summarize. Do not add events that are not spoken.

中文样本可追加：

If the speech is Chinese, transcribe it in Chinese.

6.3 Sampling

每个样本采样 4-8 个 responses，用 GRPO 相对 reward 更新。

  • group size：4 起步；显存允许再到 8。
  • max output length：严格限制，防重复。
  • temperature：0.7-1.0。
  • 保留 format invalid 样本，但 reward 置低，帮助模型学格式。

6.4 Baseline / Ablation

实验目的
Base Qwen2.5-Omni暴露问题
SFT small 1k判断纯模仿是否足够
GRPO without time reward验证 who/what 改善
GRPO full curriculum验证 when 加入后的提升

7. Demo 成功标准

在 100 条 Dialogue-TCap-Bench 上，希望看到：

指标目标
Format Pass Rate提升到 90%+
Speaker Acc相对提升 5-10 points
WER/CER相对下降 10-20%
Temporal IoU相对提升 10 points 以上
Turn F1明显提升，尤其是少合并 turn
Repetition Rate不上升，最好下降

如果 Temporal IoU 提升但 WER 变差，说明 time reward 过强。

如果 WER 提升但 speaker/time 不变，说明模型只学会 ASR，没有学会 temporal caption。

如果输出变长、重复、塞满 timestamp，说明需要更强 length/repetition penalty。

8. Demo 执行顺序

步骤动作产物/检查
Step 1造 100 条 Dialogue-TCap-Bench，覆盖 2/3/4+ turn，并人工快速检查 GT。data/dialogue_tcap_bench.json
Step 2跑 Qwen2.5-Omni baseline，统一 prompt 生成 caption。错误分析表：format、speaker、content、time、turn_merge、hallucination。
Step 3造 1000 条训练集，优先 TTS 合成 + 少量真实数据。保留 audio、turn-level transcript、speaker、start/end、prompt。
Step 4reward dry run，不训练，先对 baseline outputs 打分。检查 reward 是否区分好坏、是否符合人工判断、是否可 hack。
Step 5GRPO 训练，先 smoke test，再跑完整 1000 条。监控 mean reward、format pass、output length、repetition rate、三项 reward 曲线。
Step 6评测与 case study。Base vs GRPO 指标表 + 5-10 个 qualitative examples。

9. 风险与规避

风险表现规避
time reward 太难训练 collapse，timestamp 乱编curriculum；先 who/what 后 when
模型输出重复speaker/text 重复刷分max length；重复率惩罚；必要时 pre-DPO
speaker 匿名对齐不稳Speaker 1/2 交换导致误罚做全局 speaker permutation matching
TTS 数据过拟合真实音频无提升至少混入 300 条真实 dialogue
WER reward 压制 caption 能力模型只做 ASR，不组织 caption保持行级 speaker/time 格式 reward
重叠说话太难IoU 和 WER 都不稳定v1 丢弃 overlapped speech，v2 再加入

10. 最小可交付结果

  1. 一个 100 条的 Dialogue-TCap-Bench。
  2. 一个 1000 条的 GRPO training set。
  3. 一套 who / what / when reward。
  4. Base Qwen2.5-Omni 与 GRPO 模型的指标对比。
  5. 5-10 个能直观看出 temporal caption 改善的 case。

论文叙事可以写成：

我们发现通用 omni model 在 dialogue speech caption 中并非完全听不懂，
而是无法稳定组织 speaker、content 与 timestamp。
因此我们借鉴 D-ORCA，将 temporal caption 拆成 who/what/when 三个可验证目标，
用小规模精选数据和 curriculum GRPO 快速塑形模型的时序化 caption 行为。
